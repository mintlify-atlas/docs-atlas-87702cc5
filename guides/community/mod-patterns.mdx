Studying official DLCs and successful community mods reveals consistent patterns you can apply to your own work. This guide documents those patterns.

## Official DLC structure

Official DLCs use a **dual-folder architecture**:

- **Main content folder** (e.g., `<GAME_RESOURCES>/DLC/nepal/`) — gameplay content, assets, definitions
- **Shell folder** (e.g., `<GAME_RESOURCES>/DLC/nepal-shell/`) — metadata, dependencies, integration

This separation lets the game load metadata before committing to loading the full content.

Each DLC typically includes:
1. Civilization definitions
2. Leader definitions
3. Unique units
4. Unique infrastructure
5. Unique abilities
6. Traditions
7. Age-specific content integration
8. Localization

### DLC types

- **Civilization/Leader Packs** — complete civilizations with associated leaders (Nepal, Great Britain)
- **Natural Wonder Packs** — new wonders with placement rules
- **Alternate Leader Versions** — variations with different abilities

## Community mod example: Scythia

The Scythia mod by Izica is a well-structured example of community modding:

```text
<GAME_DATA>/Mods/civmods-izica-civilization-scythia-38056/
├── imports/
├── civilizations/
├── traditions/
├── units/
├── constructibles/
├── progression-trees/
└── izica-civilization-scythia.modinfo
```

Content is organized **by type** (not by civilization), which works well for single-civilization mods and simplifies maintenance.

## Age system integration

The Age system is the central challenge in Civ VII modding. Handle it properly from the start.

### Age assignment

- Civilizations are assigned to specific Ages
- Leaders can span Ages but may have Age-specific bonuses
- Units and buildings belong to an Age or are marked `IsAgeless="true"`

### Cross-Age persistence

Several mechanisms let content persist across Ages:

| Mechanism | How it persists |
|---|---|
| **Traditions** | Unlocked through unique Civic trees; carry over as policies |
| **Ageless Infrastructure** | Buildings/improvements with `IsAgeless="true"` |
| **Commander Units** | Persist with their attributes |

### Age transition handling

When designing mod content, consider:
- Which elements carry over to new Ages
- How deprecated elements are handled
- Whether you want to support Golden Age and Dark Age legacies

## Localization best practices

### Naming convention

Use a consistent pattern for all localization keys:

```text
LOC_MODID_CATEGORY_NAME_TEXT
```

Examples:
```text
LOC_SCYTHIA_CIVILIZATION_NAME_TEXT
LOC_SCYTHIA_LEADER_TOMYRIS_NAME_TEXT
LOC_SCYTHIA_ABILITY_DESCRIPTION_TEXT
```

### Rich text

Localization supports:
- Inline icons: `[icon:YIELD_FOOD]`
- Bold: `[B]text[/B]`
- Color coding
- Variables for dynamic content

## Structural best practices

<AccordionGroup>
  <Accordion title="Modular design">
    Organize content into logical folders by type. A clean structure makes the mod easier to maintain and debug.
  </Accordion>

  <Accordion title="Self-contained approach">
    Include all required assets and definitions within the mod. Minimize dependencies on other community mods to reduce potential conflicts.
  </Accordion>

  <Accordion title="Naming conventions">
    Use a personal or mod prefix for all identifiers:
    - Types: `UNIT_MYMOD_SCOUT` not `UNIT_SCOUT`
    - Modifiers: `MODIFIER_MYMOD_ABILITY` not `MODIFIER_ABILITY`
    - Localization keys: `LOC_MYMOD_UNIT_NAME` not `LOC_UNIT_NAME`
  </Accordion>

  <Accordion title="Dependency management">
    Explicitly declare all mod dependencies. Use `<References>` for mods that only affect load order without requiring activation.
  </Accordion>
</AccordionGroup>

## Content best practices

<AccordionGroup>
  <Accordion title="Balance">
    Design units, buildings, and abilities with the base game in mind. Test against a variety of civilizations and strategies. Avoid abilities with no meaningful counterplay.
  </Accordion>

  <Accordion title="Age compatibility">
    Assign everything to the correct Age. Test that Ageless content doesn't create unintended advantages in later Ages.
  </Accordion>

  <Accordion title="Tradition design">
    Traditions persist across Ages as policies. Design them to remain relevant across multiple historical periods, not just the Age they're introduced in.
  </Accordion>
</AccordionGroup>

## TypeScript modding tools

The community TypeScript toolkit abstracts XML manipulation and reduces common errors:

- Generates `.modinfo` and XML files from TypeScript
- Provides type-safe builder APIs for common content types
- Reduces boilerplate for complex content
- Available at: [izica/civ7-modding-tools](https://github.com/izica/civ7-modding-tools)
